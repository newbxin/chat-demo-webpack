记得。这三个说法其实是在区分“后端流里到底发什么形态的数据”，它会直接决定前端怎么渲染。

可以把它理解成 3 种不同的流式协议风格。

**1. `values` 快照**

意思是后端每次推送的不是“新增一点点内容”，而是“当前整个线程状态的最新快照”。

比如 SSE 里可能长这样：

```text
event: values
data: {
  "messages": [
    { "type": "human", "content": "你好" },
    { "type": "ai", "content": "你" }
  ],
  "title": "新对话"
}
```

过一会再推一次：

```text
event: values
data: {
  "messages": [
    { "type": "human", "content": "你好" },
    { "type": "ai", "content": "你好，我是助手" }
  ],
  "title": "新对话"
}
```

这里前后两次的区别是：

- 第二次不是只给你 `"好，我是助手"`
- 而是把“当前完整 messages”又发了一遍

前端处理方式通常是：

- 直接用新的 `messages` 覆盖旧的 `messages`

优点：
- 最稳
- 前端逻辑简单
- 不容易因为丢包、乱序、重连导致消息拼错

缺点：
- 数据量更大
- 每次都传完整快照，有重复数据

你现在这版实现就是按这个思路做的。

---

**2. 持续推送**

“持续推送”强调的是时间维度，不是数据结构本身。

它的意思是：

- 后端在生成过程中会不断往前端推数据
- 不是等全部结束后只回一次

它本身可以和“快照”结合，也可以和“增量片段”结合。

也就是说，“持续推送”回答的是：

- 数据是不是边生成边发？

而不是：

- 每次发的是完整快照，还是增量片段？

比如这两种都算“持续推送”：

1. 持续推 `values` 快照  
   每隔几百毫秒发一次完整 `messages`

2. 持续推增量 token  
   每隔几十毫秒发一个字或一个 token

所以它和前两个不是完全同一维度。

如果一定要落到前端体验上，“持续推送”意味着：

- 用户能看到答案逐步变化
- 而不是一直 loading，最后一次性出现完整答案

---

**3. 增量片段**

意思是后端每次推送的只是“新增的那一小段”，不是完整快照。

比如：

第一次：

```text
event: messages-tuple
data: { "delta": "你" }
```

第二次：

```text
event: messages-tuple
data: { "delta": "好" }
```

第三次：

```text
event: messages-tuple
data: { "delta": "，我是助手" }
```

这时前端不能直接覆盖，而要自己拼接：

```ts
existingMessage.content += delta
```

优点：
- 数据量小
- 更适合 token 级别、逐字级别显示
- 流式体验更细腻

缺点：
- 前端实现复杂很多
- 要自己处理拼接、去重、顺序、断流恢复
- 一旦漏掉某个片段，最终消息可能错

这类协议通常会出现在：

- `messages-tuple`
- `custom`
- 自定义 token 事件
- OpenAI/Anthropic 类似的 delta 流

---

**这三者的核心区别**

最容易混的是“持续推送”和另外两个。

可以这样记：

- `values` 快照：每次发“全量当前状态”
- 增量片段：每次发“新增加的那一点”
- 持续推送：描述“是不是不断地发”，不描述“发的内容结构”

所以更准确地说，常见组合其实是这几种：

1. 持续推送 + `values` 快照  
   后端不断发完整状态  
   前端不断覆盖

2. 持续推送 + 增量片段  
   后端不断发 delta  
   前端不断拼接

3. 非持续推送 + `values` 快照  
   后端最后只发一次完整结果  
   前端只能最后一次性显示

---

**为什么我之前要问你这个**

因为这会直接决定前端策略：

如果是 `values` 快照：
- 前端最简单
- `setThreadState(messages = values.messages)` 就行

如果是增量片段：
- 前端要维护“当前正在生成的 AI message”
- 每次把 delta 拼进去
- 还要考虑 tool call、reasoning、附件、重连等复杂情况

如果只是最终一次返回：
- 那就不是真正意义上的“流式渲染”
- 前端只能显示 loading，结束后整体替换

**结合你当前这版项目**

你现在这套代码默认假设的是：

- 后端会持续推送
- 推送的主事件是 `event: values`

而不是自己逐 token 拼接。

**一句话总结**

三者的本质区别是：

- `values` 快照：发“全量”
- 增量片段：发“增量”
- 持续推送：发“一次还是多次”
