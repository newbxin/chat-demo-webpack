`optimisticMessages` 是“乐观消息”的本地状态。

它的作用是：后端真实结果还没回来之前，先把用户刚发出去的消息，或者附件上传中的占位提示，立刻显示在界面上，避免用户感觉“点了发送没反应”。

在 [hooks.ts](/d:/github/chat-demo-webpack/src/core/threads/hooks.ts) 里它主要做两件事：

1. 先展示用户消息  
   用户一提交，`sendMessage` 就会先塞一个 `human` 类型的 optimistic message 到 `optimisticMessages`，这样 UI 不用等接口返回就能马上看到这条消息。

2. 附件上传时展示过渡状态  
   如果有文件，会额外放一个临时的 AI message，比如 `"Uploading files, please wait..."`，提示当前还在上传。

后面一旦 SSE 返回了真正的 `values.messages`，代码就会：

- `setOptimisticMessages([])`

把这些临时消息清掉，因为这时候已经有服务端权威消息快照了，不需要前端占位了。

你可以把它理解成：

- `threadState.messages`：服务端已经确认的真实消息
- `optimisticMessages`：前端为了提升体验临时加上的消息
- 最后渲染时，两者会合并成 `mergedThread.messages`

也就是这里的思路：

```ts
messages: [...threadState.messages, ...optimisticMessages]
```

所以它本质上是一个“请求结果回来前的 UI 缓冲层”。

可以，我们按 [hooks.ts](/d:/github/chat-demo-webpack/src/core/threads/hooks.ts) 里每一次 `setOptimisticMessages(...)` 来看它分别在干什么。

**1. 切换线程时清空**
在 `useEffect([threadId])` 里：

```ts
setOptimisticMessages([]);
```

作用：
当前线程变了，就把上一个线程的临时消息清掉，避免旧线程发送中的占位消息串到新线程页面里。

**2. 刚发送时先放入乐观消息**
在 `sendMessage` 一开始：

```ts
setOptimisticMessages(nextOptimisticMessages);
```

这里会放进去的通常是：

- 一条临时的用户消息 `human`
- 如果有附件，再加一条临时 AI 提示消息，比如“Uploading files...”

作用：
用户点发送后，界面立刻有反馈，不需要等接口返回。

**3. 文件上传成功后，更新乐观消息里的文件状态**
在附件上传完成后：

```ts
setOptimisticMessages((prev) => {
  ...
  return [
    {
      ...prev[0],
      additional_kwargs: { files: uploadedFiles },
    },
    ...prev.slice(1),
  ];
});
```

作用：
把第一条临时用户消息里的文件信息，从“上传中”改成“已上传”，并补上真实的 `path / virtual_path` 等信息。

也就是：
不是新增一条消息，而是“更新已有的 optimistic 用户消息”。

**4. 收到服务端 `values` 快照后清空**
在 SSE 事件处理里，收到 `event: values` 时：

```ts
setOptimisticMessages([]);
```

作用：
后端已经返回了真实的 `messages` 快照，这时候前端临时占位消息就该退场了，否则会和服务端真实消息重复。

这是最关键的一次清空。

**5. 请求失败时清空**
在 `catch` 里：

```ts
setOptimisticMessages([]);
```

作用：
如果创建线程、上传文件、发起流式请求等任何一步失败，就把临时占位消息移除，避免页面一直挂着一条“假的已发送消息”或“上传中”提示。

**一句话总结**

这几次 `setOptimisticMessages` 分别对应 4 类动作：

- 进入新线程时清场
- 发送瞬间先占位
- 上传完成后修正占位内容
- 服务端真实数据回来或失败时撤掉占位

你可以把它理解成一个完整生命周期：

`创建临时消息 -> 更新临时消息 -> 被真实消息替换 / 被失败清掉`
