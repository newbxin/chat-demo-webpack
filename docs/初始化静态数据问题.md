## 根因分析

问题是一个 **循环触发链**：

1. `ChatDemo` 中 `useEffect` 调用 `sessionDispatch.setThreadState(nextThreadState)` → 更新 `MainProvider` 中的 `state`
2. `MainProvider` 的 `state` 变化 → 重新渲染 → `contextValue` 变化 → `ChatDemo` 重新渲染
3. `ChatDemo` 重新渲染 → `currentThread`（对象字面量）是新引用 → `useMemo` 重新计算 → `nextThreadState` 是新引用
4. `nextThreadState` 变化 → `useEffect` 再次触发 → 回到第1步

你之前用 `useMemo` 包裹 `currentThread` 后不显示数据，是因为 **`threadData` 来自静态导入的 `threads` 数组，引用始终不变**，所以 `useMemo` 只在首次渲染计算一次。但此时 `MainProvider` 的 reducer 初始状态是空的（`createThreadStreamState(null, null)`），`useEffect` 只执行一次就停了——这本身是对的。

问题可能在于你修改后 `useEffect` 的依赖写法，或者 `sessionDispatch` 没被包含在依赖中。让我确认一下你的当前代码。

## 正确的修复方案

其实根本不需要这个 `useEffect`。既然 `threads` 是静态数据，应该直接通过 `MainProvider` 的 `initialState` 传入，让 Provider 内部的 `useEffect`（[SessionProvider.tsx:394-402](src/providers/SessionProvider.tsx#L394-L402)）来处理初始化：

```tsx
// App.tsx
export default function App() {
  const threadData = threads[0];
  const initialState = threadData ? {
    ...threadData.data,
    isLoading: false,
    isThreadLoading: false,
    messages: threadData.data.values.messages as Message[],
  } : null;

  return (
    <MainProvider initialThreadId={threadData?.id} initialState={initialState}>
      <ChatDemo />
    </MainProvider>
  );
}
```

然后在 `ChatDemo` 中**去掉** `useEffect` 和 `nextThreadState` 相关逻辑，直接使用 `session.thread`：

```tsx
export function ChatDemo() {
  const currentThreadId = threads[0]?.id || '';
  const [settings] = useLocalSettings();
  const [session, sessionDispatch] = useSessionContext(SessionType.main);
  // 不再需要 useEffect + setThreadState，数据已经通过 MainProvider 初始化
  // ...
}
```

这才是 `MainProvider` 的 `initialState` 和 `initialThreadId` 参数的设计意图——在 Provider 层完成初始化，而不是在子组件中手动 `setThreadState`。
