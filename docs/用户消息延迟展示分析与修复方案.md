# 用户消息延迟展示分析与修复方案

## 现象

当前聊天页中，用户点击发送后，`message-list` 不会立刻显示这条用户消息，而是要等一小段时间，通常在后端流式返回首个事件后才出现。

## 结论

根因不在 `MessageList` 渲染层，而在发送链路的数据归属上：

- 前端虽然创建了 `optimisticHumanMessage`
- 但这条消息默认只进入 `optimisticMessages`
- 它没有立即写入 `threadState.messages`
- 后续一旦收到 `values` 事件，又会先清空 `optimisticMessages`
- 然后直接用后端返回的 `values.messages` 覆盖主消息列表

这会导致“用户消息是否显示、何时显示”被绑定到后端 `values` 事件的时机与内容上。

## 关键代码定位

### 1. optimistic human message 只进入临时展示层

文件：`src/core/threads/hooks.ts`

- `optimisticHumanMessage` 在发送时被创建
- 但只进入 `nextOptimisticMessages`
- 然后通过 `sessionDispatch.setOptimisticMessages(...)` 写入临时状态

这意味着它不是主线程消息的一部分。

### 2. 渲染时只是把主消息和 optimistic 消息拼接

文件：`src/providers/SessionProvider.tsx`

当前 `thread` 的构造方式是：

- 主体来自 `state.threadState.messages`
- 如果存在 `state.optimisticMessages`，则做数组拼接

也就是说 optimistic 数据只是一个“覆盖层”，并不稳定。

### 3. 收到 `values` 事件后会立即清掉 optimistic

文件：`src/core/threads/hooks.ts`

在 `event.event === "values"` 分支里，当前逻辑会：

1. `setOptimisticMessages([])`
2. `applyThreadSnapshot(prev, event.data)`

而 `applyThreadSnapshot` 默认把 `values.messages` 视为权威快照并直接替换当前消息列表。

### 4. 当前机制对后端快照契约过于敏感

如果后端首个 `values.messages`：

- 没有包含刚发送的 human message
- 或返回较晚
- 或只是部分 AI 消息快照

那么前端就会出现：

- 用户消息展示延迟
- 用户消息闪一下又消失
- 本地 optimistic 与后端快照互相覆盖

## 问题本质

现在的实现把“用户消息立即展示”建立在了临时 optimistic overlay 上，而不是建立在线程主状态上。

这会带来两个问题：

1. 用户消息不是线程主状态的一部分，稳定性不足
2. 首个 `values` 事件到来后，前端会把展示权重新交回后端快照

因此只要后端快照稍慢或不完整，用户就会感知到消息出现延迟。

## 修复目标

修复后的行为应当是：

1. 用户点击发送后，human message 立即写入 `threadState.messages`
2. UI 在本地先展示这条消息，不依赖首个 `values` 事件
3. 后端 `values` 到来后，对主消息列表做对齐与补全，而不是粗暴覆盖
4. 临时 optimistic 状态只用于“上传中附件占位 / 辅助占位 UI”，不再承担用户消息本体

## 修改方案

### 方案一：用户消息直接进入主状态

在发送入口中：

- 构造 `optimisticHumanMessage`
- 立即写入 `threadState.messages`
- 同时同步 `values.messages`

这样 `MessageList` 读取到的主消息列表会立刻包含这条用户消息。

### 方案二：只保留必要的 optimistic overlay

`optimisticMessages` 后续只用于：

- 文件上传中的占位 AI 消息
- 其他纯展示型临时节点

不再承载用户真正发送的 human message。

### 方案三：`values` 事件改为 reconcile，而不是 replace

收到后端快照时：

- 优先用后端消息更新已有消息
- 对已有的本地 optimistic human message 做匹配替换
- 如果后端还没带回这条 human message，就先保留本地这条消息

这样可以兼容：

- 后端返回完整快照
- 后端只返回部分消息
- 后端消息 ID 与本地 optimistic ID 不一致

## 本次准备采用的最小改动

本次修复先采用最小可落地方案：

1. 修改 `src/core/threads/hooks.ts`
2. 用户发送后立即把 human message 写入 `threadState.messages`
3. 对 `values.messages` 做增量对齐，而不是直接替换
4. 保留 `optimisticMessages` 作为上传占位层

## 预期效果

修复后，用户点击发送时应看到：

1. 用户消息立即出现在列表中
2. 即使后端首个 `values` 返回较慢，用户消息也不会延迟展示
3. AI 流式内容继续按现有 SSE 逻辑更新
4. 不会因为后端快照缺少 human message 而把本地消息冲掉
